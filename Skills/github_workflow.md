# GitHub Workflow Manager

> 팀 협업 환경에서의 안전한 버전 관리 및 코드 통합을 보장하기 위한 단계별 GitHub 연동 파이프라인 명세입니다.

---

## 1. 정의 및 역할 (Definition & Role)
- **버전 관리 표준화:** 기능 분기, 코드 커밋, 원격 저장소 동기화 및 병합 과정에서 발생할 수 있는 충돌을 예방하고 팀의 형상 관리 규칙을 준수합니다.
- **오류 방지:** 강제 푸시 차단, 민감 정보 노출 방지 등 안전 장치를 통해 원격 저장소의 무결성을 보호합니다.
- **작업 파이프라인 제어:** 단일 커맨드에 의한 무분별한 덮어쓰기를 방지하고, 검증-분기-커밋-공유 단계로 세분화하여 AI의 작업을 통제합니다.

## 2. 핵심 실행 프로세스 (Execution Pipeline)

### Phase 1: 상태 검증 및 동기화 (Status Check & Safe Pull)
1. **작업 디렉토리 확인:** `git status`를 통해 로컬 변경 사항 유무를 확인합니다. 미반영 변경 사항이 있을 경우 `git stash`로 임시 저장합니다.
2. **원격 동기화:** `git fetch` 및 `git pull --rebase origin [branch]`를 실행하여 머지 커밋(Merge Commit) 없이 최신 원격 코드로 로컬을 갱신합니다.
3. **상태 복원:** 필요 시 `git stash pop`으로 기존 작업 내용을 복구하고 충돌(Conflict) 여부를 확인합니다.

### Phase 2: 브랜치 분기 (Branching Strategy)
1. **브랜치 생성:** 새로운 기능 추가나 버그 수정 시 `main` 브랜치에 직접 작업하지 않고, 명명 규칙(예: `feat/`, `fix/`, `chore/`)을 준수하여 새 브랜치를 생성합니다. (`git checkout -b [branch_name]`)
2. **작업 격리:** 모든 변경 사항은 격리된 브랜치 내에서 수행되어야 합니다.

### Phase 3: 원자적 단위 커밋 (Atomic Commit)
1. **변경 사항 분리:** `git add .` 사용을 지양하고, 논리적 단위로 연관된 파일만 선택적으로 스테이징(`git add [file_path]`)합니다.
2. **규격화된 커밋 메시지:** Conventional Commits 규격(예: `feat: [기능 요약]`)을 따라 커밋 메시지를 작성하여 작업 이력을 명확히 남깁니다.

### Phase 4: 리뷰 요청 및 공유 (Push & PR)
1. **원격 푸시:** `git push origin [branch_name]`을 통해 원격 저장소에 작업 브랜치를 업로드합니다.
2. **PR(Pull Request) 생성:** GitHub 도구를 활용해 팀원에게 리뷰를 요청하는 PR을 생성하며, 변경 목적과 영향을 상세히 기재합니다.

## 3. 기술적 제약 사항 (Constraints)
1. **강제 푸시(Force Push) 금지:** `main`, `master` 등 공용 브랜치에서는 어떠한 경우에도 `git push --force` 등 강제 덮어쓰기 명령어를 사용할 수 없습니다.
2. **민감 정보 하드코딩 검증:** 커밋을 수행하기 전, API Key, 패스워드, 로컬 시스템의 절대 경로 등 보안 위협 요소가 포함되어 있는지 반드시 확인합니다.
3. **독립적 프로세스 유지:** 각 Phase는 독립적으로 실행되며, 이전 단계의 무결성(예: 머지 충돌 없음)이 확보된 이후에만 다음 단계의 Git 명령을 수행합니다.
